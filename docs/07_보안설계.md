# 07. 보안 설계

## 1. 인증 방식 (Authentication)

### JWT (JSON Web Token) 기반 인증

```
로그인 요청
    → ID/PW 검증 (BCrypt)
    → Access Token (1시간) + Refresh Token (7일) 발급
    → 이후 요청: Authorization: Bearer {token} 헤더 포함
    → JwtAuthenticationFilter에서 토큰 검증
    → SecurityContext에 인증 정보 저장
```

### 토큰 구성

| 토큰 | 유효기간 | 저장 위치 | 용도 |
|------|----------|-----------|------|
| **Access Token** | 1시간 | 브라우저 메모리 또는 HttpOnly Cookie | API 인증 |
| **Refresh Token** | 7일 | Redis (서버) + HttpOnly Cookie | Access Token 재발급 |

### JWT Payload 구성
```json
{
  "sub": "1",
  "loginId": "admin",
  "userName": "관리자",
  "roles": ["ROLE_ADMIN"],
  "iat": 1711756800,
  "exp": 1711760400
}
```

## 2. 인가 방식 (Authorization)

### 역할 기반 접근 제어 (RBAC)

```
사용자 ──── (N:M) ──── 역할 ──── (N:M) ──── 메뉴/권한
```

### 기본 역할 (Role) 정의

| 역할 코드 | 역할명 | 접근 범위 |
|-----------|--------|-----------|
| `ROLE_ADMIN` | 시스템 관리자 | 전체 메뉴 접근 |
| `ROLE_HR` | 인사 담당자 | 인사/근태/급여 모듈 |
| `ROLE_ACCOUNTING` | 재무 담당자 | 회계/예산 모듈 |
| `ROLE_PURCHASE` | 구매 담당자 | 구매/재고 모듈 |
| `ROLE_SALES` | 영업 담당자 | 영업/재고 모듈 |
| `ROLE_USER` | 일반 사용자 | 개인 근태/휴가 |

### URL 레벨 권한 설정 (SecurityConfig)
```java
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/v1/auth/**").permitAll()
    .requestMatchers("/api/v1/system/**").hasRole("ADMIN")
    .requestMatchers("/api/v1/hr/**").hasAnyRole("ADMIN", "HR")
    .requestMatchers("/api/v1/accounting/**").hasAnyRole("ADMIN", "ACCOUNTING")
    .requestMatchers("/api/v1/purchase/**").hasAnyRole("ADMIN", "PURCHASE")
    .anyRequest().authenticated()
);
```

### 메서드 레벨 권한 설정
```java
@PreAuthorize("hasRole('HR')")
public EmployeeResponse createEmployee(EmployeeRequest request) { ... }

@PreAuthorize("hasAnyRole('ADMIN', 'HR')")
public Page<EmployeeResponse> findEmployees(SearchDto dto, Pageable pageable) { ... }
```

## 3. 비밀번호 정책

| 항목 | 정책 |
|------|------|
| **최소 길이** | 8자 이상 |
| **구성 요소** | 영문 + 숫자 + 특수문자 조합 |
| **암호화** | BCrypt (strength=12) |
| **유효기간** | 90일 (만료 시 변경 강제) |
| **이전 비밀번호** | 최근 3개 재사용 금지 |
| **초기 비밀번호** | 임시 발급 후 최초 로그인 시 변경 강제 |

## 4. 세션 보안

| 항목 | 정책 |
|------|------|
| **세션 저장소** | Redis (서버 사이드) |
| **세션 타임아웃** | 30분 (무활동 기준) |

| **세션 고정 공격 방지** | 로그인 시 세션 ID 재발급 |

## 5. 입력값 검증 (Validation)

```java
// DTO 레벨 검증
public class EmployeeRequest {
    @NotBlank(message = "사번은 필수입니다")
    @Size(max = 20, message = "사번은 20자 이내")
    private String empNo;

    @NotBlank(message = "이름은 필수입니다")
    @Size(max = 100)
    private String empName;

    @Email(message = "이메일 형식이 올바르지 않습니다")
    private String email;

    @NotNull(message = "입사일은 필수입니다")
    @PastOrPresent(message = "입사일은 현재 또는 과거 날짜")
    private LocalDate hireDate;
}
```

## 6. XSS / CSRF 방지

| 보안 위협 | 대응 방안 |
|-----------|-----------|
| **XSS** | Thymeleaf 자동 이스케이프, 입력값 HTML 태그 제거 |
| **CSRF** | REST API (Stateless) + JWT → CSRF 토큰 불필요 |
| **SQL Injection** | JPA/QueryDSL 파라미터 바인딩 사용 (직접 쿼리 금지) |
| **Path Traversal** | 파일 업로드 경로 검증 |

## 7. 감사 로그 (Audit Log)

모든 데이터 변경 이력을 `sys_audit_log` 테이블에 기록합니다.

```sql
CREATE TABLE sys_audit_log (
    log_id       BIGSERIAL    PRIMARY KEY,
    table_name   VARCHAR(100) NOT NULL,   -- 변경된 테이블
    record_id    BIGINT,                  -- 변경된 레코드 PK
    action       VARCHAR(10)  NOT NULL,   -- INSERT/UPDATE/DELETE
    changed_data JSONB,                   -- 변경 전/후 데이터 (JSON)
    user_id      BIGINT,
    user_ip      VARCHAR(45),
    created_at   TIMESTAMP    NOT NULL DEFAULT NOW()
);
```

### Spring AOP 기반 감사 처리
```java
@Aspect
@Component
public class AuditAspect {
    @AfterReturning(
        pointcut = "@annotation(Auditable)",
        returning = "result"
    )
    public void logAudit(JoinPoint jp, Object result) {
        // 감사 로그 저장 처리
    }
}
```

## 9. HTTPS 설정

- **개발환경**: HTTP 허용
- **운영환경**: HTTPS 필수 (Nginx에서 SSL 종료)
- **HSTS**: `Strict-Transport-Security` 헤더 적용
- **인증서**: Let's Encrypt 또는 사내 인증서
