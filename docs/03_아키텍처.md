## 1. 전체 구조 (Overview)

```

  │  ─────────────────────────────────────────  │

  │  ─────────────────────────────────────────  │

  │  논블로킹 DB 쿼리 (ReactiveCrudRepository)  │
  └─────────────────────────────────────────────┘
         │                         │
         ▼                         ▼

  논블로킹 커넥션              논블로킹 캐시/세션
```

## 4. 요청 처리 흐름 (Reactive)

### REST API 흐름
```
브라우저 POST /api/v1/hr/employees (JSON)
    │
    ▼ [논블로킹 I/O - Netty가 이벤트 루프로 처리]
    │
    ▼ JwtAuthFilter (WebFilter) — JWT 검증 → SecurityContext 설정
    │
    ▼ @RestController.create(Mono<EmployeeRequest>)
    │   └─ @Valid 검증
    │
    ▼ EmployeeService.create(request)
    │   ├─ existsByEmpNo() → Mono<Boolean>     [논블로킹 DB 조회]
    │   └─ save(entity)    → Mono<Employee>    [논블로킹 DB INSERT]
    │
    ▼ map(entity → EmployeeResponse)            [동기 변환]
    │
    ▼ Mono<ApiResponse<EmployeeResponse>> 반환
    │
    ▼ Netty가 HTTP 응답 비동기 전송
```

### Operator 체이닝 예시
```java
// 직원 등록 서비스 흐름 (operator 체이닝)
return employeeRepository.existsByEmpNo(request.getEmpNo())  // 1. 사번 중복 확인
    .flatMap(exists -> exists                                  // 2. 중복이면 에러
        ? Mono.error(new ErpException(ErrorCode.DUPLICATE_EMP_NO))
        : employeeRepository.save(Employee.create(request))   // 3. 저장
    )
    .map(EmployeeResponse::from)                              // 4. DTO 변환
    .doOnSuccess(res -> log.info("등록 완료: {}", res.getEmpId())); // 5. 로그
```

## 5. 보안 아키텍처 (Reactive)

```
HTTP 요청
    │
    ▼
JwtAuthenticationWebFilter          ← WebFilter (논블로킹 JWT 검증)
    │ Mono<Authentication>
    ▼
ReactiveSecurityContextHolder        ← 리액티브 보안 컨텍스트 저장
    │
    ▼
@PreAuthorize("hasRole('HR')")       ← 메서드 레벨 권한 검사 (Reactive 지원)
    │
    ▼
Controller → Service → Repository
```

> ℹ️ Spring MVC의 `SecurityContextHolder.getContext()` 대신
> `ReactiveSecurityContextHolder.getContext()` 를 사용합니다.
> (ThreadLocal 대신 Reactor Context 기반)

## 6. 스레드 모델 비교

### Spring MVC (Blocking)
```
요청 1 → [Thread-1] → DB 대기(블로킹)... → 응답
요청 2 → [Thread-2] → DB 대기(블로킹)... → 응답
요청 3 → [Thread-3] → DB 대기(블로킹)... → 응답
...
요청 N → ❌ 스레드 없음 → 대기 (성능 저하)
```

### Spring WebFlux (Non-Blocking)
```
요청 1 ─┐
요청 2 ─┤─→ [EventLoop Thread] → DB 완료 이벤트 수신 → 응답 1
요청 3 ─┤─→ [EventLoop Thread] → 다른 작업 처리    → 응답 2
요청 N ─┘                                            → 응답 N
(소수의 스레드로 수천 개의 동시 요청 처리 가능)
```

## 7. 캐시 전략 (Redis Reactive)

```java
// Redis Reactive 캐시 예시
public Mono<List<CommonCode>> getCommonCodes(String codeGroup) {
    String cacheKey = "common-code:" + codeGroup;

    return redisTemplate.opsForValue().get(cacheKey)  // 1. 캐시 조회
        .switchIfEmpty(                                 // 2. 캐시 없으면
            commonCodeRepository.findByCodeGroup(codeGroup)
                .collectList()                          // 3. DB 조회
                .flatMap(codes ->
                    redisTemplate.opsForValue()
                        .set(cacheKey, codes, Duration.ofHours(1))  // 4. 캐시 저장
                        .thenReturn(codes)              // 5. 결과 반환
                )
        );
}
```

| 데이터 | 캐시 정책 | TTL |
|--------|-----------|-----|
| 공통코드 | Cache-Aside (Reactive) | 1시간 |
| 메뉴/권한 | Cache-Aside (Reactive) | 30분 |
| 세션 | Write-Through | 30분 (자동 연장) |
