### DB/계정 생성 SQL (root 계정으로 실행)

```sql
-- 1. 데이터베이스 생성
CREATE DATABASE IF NOT EXISTS eterp
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_unicode_ci;

-- 2. 계정 생성 (외부 접속 허용)
CREATE USER IF NOT EXISTS 'eterp'@'%' IDENTIFIED BY 'eterp1234!';

-- 3. 권한 부여
GRANT ALL PRIVILEGES ON eterp.* TO 'eterp'@'%';

-- 4. 권한 적용
FLUSH PRIVILEGES;

-- 5. 확인
SHOW DATABASES;
SELECT user, host FROM mysql.user WHERE user = 'eterp';
```

> ⚠️ **비밀번호** `eterp1234!` 는 예시입니다. 실제 사용할 강한 비밀번호로 바꾸세요!

### 기존 MVC → WebFlux 전환 변경사항

| 항목 | Spring MVC (기존) | Spring WebFlux (변경) |
|------|-------------------|-----------------------|
| **웹 프레임워크** | Spring MVC | Spring WebFlux |
| **서버** | Embedded Tomcat | Embedded Netty (논블로킹) |
| **DB 접근** | Spring Data JPA (블로킹) | Spring Data R2DBC (논블로킹) |
| **DB 드라이버** | JDBC | R2DBC |
| **반환 타입** | `Employee`, `List<Employee>` | `Mono<Employee>`, `Flux<Employee>` |
| **보안** | Spring Security (Servlet) | Spring Security Reactive |
| **트랜잭션** | `@Transactional` | `@Transactional` (Reactive 지원) |

### build.gradle.kts 변경사항

```kotlin
dependencies {
    // WebFlux (MVC 대신 사용 - Netty 내장 서버)
    implementation("org.springframework.boot:spring-boot-starter-webflux")

    // Thymeleaf + Reactive
    implementation("org.springframework.boot:spring-boot-starter-thymeleaf")
    implementation("org.thymeleaf.extras:thymeleaf-extras-springsecurity6")

    // R2DBC (Reactive DB 접근 - JPA 대신)
    implementation("org.springframework.boot:spring-boot-starter-data-r2dbc")

    // MariaDB R2DBC 드라이버
    runtimeOnly("org.mariadb:r2dbc-mariadb:1.2.2")
    // 마이그레이션용 JDBC
    runtimeOnly("org.mariadb.jdbc:mariadb-java-client:3.5.1")

    // Spring Security Reactive
    implementation("org.springframework.boot:spring-boot-starter-security")

    // Reactor 테스트 도구
    testImplementation("io.projectreactor:reactor-test")

    runtimeOnly("io.jsonwebtoken:jjwt-impl:0.12.6")
```

### 리액티브 Controller 예시 (한글 주석)

```java
/**
 * 직원 관리 컨트롤러 (Reactive)
 * - 모든 메서드는 Mono 또는 Flux를 반환합니다.
 * - block(), blockFirst() 등 블로킹 메서드는 절대 사용하지 마세요!
 */
@RestController
@RequestMapping("/api/v1/hr/employees")
@RequiredArgsConstructor
@Slf4j
public class EmployeeController {

    private final EmployeeService employeeService;

    @GetMapping
    public Flux<EmployeeResponse> findAll() {
        return employeeService.findAll();
    }

    @GetMapping("/{empId}")
    public Mono<EmployeeResponse> findById(@PathVariable Long empId) {
        return employeeService.findById(empId);
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public Mono<EmployeeResponse> create(
            @Valid @RequestBody Mono<EmployeeRequest> requestMono) {
        return requestMono.flatMap(employeeService::create);
    }

    @PutMapping("/{empId}")
    public Mono<EmployeeResponse> update(
            @PathVariable Long empId,
            @Valid @RequestBody Mono<EmployeeRequest> requestMono) {
        return requestMono.flatMap(req -> employeeService.update(empId, req));
    }

    @DeleteMapping("/{empId}")
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public Mono<Void> delete(@PathVariable Long empId) {
        return employeeService.delete(empId);
    }
}
```

## Q15. 인증 모듈(Auth)의 범위: 회원가입 vs 일반 로그인

| 항목 | 내용 |
|------|------|
| 🙋 **질문일시** | 2026-03-30 14:06 |
| 🤖 **답변일시** | 2026-03-30 14:08 |

**질문:**
> 일반적으로 로그인 회원가입 기능을 개발 한다는거지?

**답변:**

맞습니다! 하지만 **사내 ERP 시스템**이라는 특성상 일반적인 웹사이트(B2C)의 인증 모듈과는 조금 다르게 접근합니다.

> 즉, lgcns 팀이면 lgcns 팀 사원과 팀원 공지, 전체 공지만 보이고, lguplus 팀이면 해당 팀원만 보여야 해. 이해가 됨?

**답변:**

네, 완벽하게 이해했습니다! 매우 실무적이고 중요한 요구사항입니다.

시스템의 **권한(Role)** 과 **데이터 접근 범위(Data Visibility)** 를 명확히 분리하여 설계해야 합니다.

### 1. 권한(Role) 영어 명칭 정의
스프링 시큐리티 표준에 맞춰 `ROLE_` 접두사를 사용해 정의합니다.

- **admin (최고 관리자)**: `ROLE_ADMIN` (모든 데이터 접근 및 시스템 설정 권한)
- **관리자 (중간 관리자/인사담당)**: `ROLE_MANAGER` (모든 팀의 데이터 조회 및 승인 가능)
- **팀장**: `ROLE_TEAM_LEADER` (자신의 속한 부서/팀의 데이터만 관리 및 승인)
- **팀원**: `ROLE_TEAM_MEMBER` (자신의 소속 부서 데이터만 조회, 본인 데이터만 수정/신청)
- **프리 (프리랜서)**: `ROLE_FREELANCER` (팀원과 유사하나 계약직 관리에 따른 최소한의 데이터만 접근)

### 2. 부서별 데이터 격리 (Data Isolation) 로직 적용
팀장(`ROLE_TEAM_LEADER`), 팀원(`ROLE_TEAM_MEMBER`), 프리(`ROLE_FREELANCER`)가 로그인할 때는 다음 규칙이 적용됩니다.

- **JWT 토큰에 부서 정보 포함**: 로그인 시 발급되는 JWT Access Token에 사용자의 `ROLE` 뿐만 아니라 소속된 **`deptId` (부서 ID)** 를 함께 저장합니다.
- **R2DBC Repository 조회 쿼리 조건 처리**:
  API 오퍼레이션이 일어날 때마다(예: 사원 목록 조회, 근태 현황 조회, 다이어리 조회 등) SecurityContext에서 사용자의 토큰 정보를 꺼내옵니다.
  - 사용자가 `ROLE_ADMIN` 또는 `ROLE_MANAGER` 이면 부서 조건 없이 전체 결과를 반환합니다.
  - 사용자가 그 외 권한(`ROLE_TEAM_LEADER`, `ROLE_TEAM_MEMBER` 등)이면 쿼리의 `WHERE` 절에 무조건 `AND dept_id = :deptId` 문구를 덧붙여 **오직 자신이 속한 부서 데이터만 조회**되도록 서버단에서 강력하게 강제합니다.
  - **공지사항(Notice)** 의 경우, `공지 대상(target_dept_id)`이 '전체(`NULL` 또는 `ALL`)'이거나 '내 부서 ID'와 일치하는 데이터만 불러오도록 쿼리를 작성합니다.

이렇게 구조를 잡으면 lgcns 팀 소속 사용자는 절대로 lguplus 팀의 데이터를 화면과 API 단에서 조회할 수 없게 됩니다.

위 내용에 동의하시면 코드 상에 **권한 Enum** 객체를 생성하고, JWT에 **`deptId`** 를 담는 로직부터 구현 시작하겠습니다!

## Q20. Git에 .java 소스만 올라가고 .class는 올리지 않는 이유

| 항목 | 내용 |
|------|------|
| 🙋 **질문일시** | 2026-03-30 14:35 |
| 🤖 **답변일시** | 2026-03-30 14:38 |

**질문:**
> git 에서 업로드 할때는 java까지 업로드가 필요하지만 git pull 받을때 굳이 자바파일은 필요 없고 .class 파일만 있으면 되는거 아니야?

**답변:**

매우 예리한 질문입니다! 서버에서 단순히 실행만 할 목적이라면 `.class` 파일(또는 빌드된 `.jar` 파일)만 있으면 되는 것이 맞습니다. 하지만 **Git의 목적** 때문에 다음과 같이 처리합니다.

**1. Git은 "소스 코드" 버전 관리 도구입니다.**
- Git은 개발자 간의 텍스트 기반 소스코드(`.java`, `.html`, `.yml`)를 공유하고 변경 이력을 추적하는 도구입니다.
- `.class` 파일은 코드를 컴파일하면 언제든 기계(컴퓨터)가 자동으로 만들어낼 수 있는 결과물(바이너리)이므로 파일 용량만 차지할 뿐 Git으로 관리하지 않습니다. (그래서 우리가 미리 설정한 `.gitignore` 파일에 `*.class` 와 `build/` 폴더가 등록되어 있어 업로드가 차단됩니다.)

**2. 우분투 서버에서의 실행 방식**
- 우분투 서버에서 `git pull`을 받으면 윈도우와 똑같은 `.java` 소스코드가 다운로드됩니다.
- 서버에서 `./gradlew bootRun` 또는 `./gradlew build` 명령어를 실행하면, **우분투 서버에 설치된 자바(JDK)가 `.java` 파일을 읽어서 실시간으로 `.class` 파일로 컴파일**한 뒤 서버를 띄우게 됩니다.

**3. 만약 운영(Production) 서버라면?**
- 운영 서버에는 보통 Git을 설치하거나 소스코드를 다운로드 받지 않습니다.
- 윈도우 개발 PC나 CI/CD(Jenkins 등) 서버에서 소스코드를 컴파일하여 `.jar` 단일 파일(모든 `.class`가 압축된 실행 파일) 하나만 만든 다음, 이 **`.jar` 파일만 FTP 통신 방식 등을 통해 운영 서버로 전송**하여 실행하는 것이 정석입니다.

지금은 우분투 장비를 **"테스트용 개발 서버"** 로 활용하고 계시기 때문에 소스 단위로 Git 동기화를 하는 것입니다!

`/home/a86223/ws/etErp` 경로에서 사용할 수 있도록 최상단에 `run.sh` 파일을 만들었습니다. 제공하는 명령어는 다음과 같습니다:

1. `./run.sh rebuild` 
   - **가장 많이 쓰실 명령어입니다.** 
   - (1) 현재 띄워진 서버를 안전하게 종료(`stop`)
   - (2) `git pull origin main` 으로 윈도우에서 방금 짠 최신 코드를 다운로드
   - (3) `gradlew clean build` 로 최신 `.class` 파일 생성
   - (4) 백그라운드(`nohup`)로 애플리케이션 다시 띄우기(`start`)
   - 이 모든 4단계를 명령어 하나로 처리합니다.

2. `./run.sh build`: Pull 제외, 순수하게 빌드만 수행
3. `./run.sh start`: 빌드된 기존 버전을 백그라운드로 실행
4. `./run.sh stop`: 현재 백그라운드에서 돌아가는 ERP 프로세스 찾아 종료 처리
5. `./run.sh status`: 서버가 죽었는지 안 죽었는지 확인
6. `./run.sh log`: 실시간 로그(`application.log`) 감시 모드로 들어가기 (종료는 `Ctrl+C`)

**[중요] 우분투에서 최초 1회 해상 스크립트에 "실행 권한"을 부여해 주셔야 합니다.**
```bash
# git pull 로 run.sh 파일을 다운받은 직후 한 번만 수행
chmod +x run.sh
```
이후부터는 자유롭게 `./run.sh rebuild` 나 `./run.sh log` 등을 사용하시면 윈도우와 우분투 간 연동 속도가 비약적으로 빨라질 것입니다!

| 항목 | 내용 |
|------|------|
| 🙋 **질문일시** | 2026-03-30 14:58 |

**질문:**
> 현재 나는 ledmega.cloud 도메인을 운영중이고 서버는 2개 돌아가고 있거든
> 프록시(8080) / openclaw(18789)
> 다른 포트로 돌려야 하겠지? Nginx 설정은 어떻게 해?

**답변:**

정확합니다! 현재 호스트의 `8080` 포트를 기존 mega Spring Boot가 선점하고 있으므로, etERP 서버는 무조건 다른 포트를 써야 합니다. 

**상황 해결 1. Spring Boot(etErp)의 자체 포트 8081로 변경**
- 충돌을 피하기 위해 `application.yml` 파일에서 `server.port: 8081` 로 설정을 변경했습니다. (현재 GitHub에 Push 완료)
- 이제 Spring Boot는 `172.17.0.1:8081` 로 구동됩니다.

**상황 해결 2. Nginx 설정 (`default.conf`) 추가 방법 제안**
사용자 입장에서 접속을 가장 편하게 하려면, 새로운 포트를 여는 것보다 **서브도메인(`erp.ledmega.cloud`)** 을 사용하는 첫 번째 방식을 가장 추천합니다. (기존 Nginx 파일 하단에 아래 `server` 블록을 통째로 추가하시면 됩니다)

**👉 방법 A (추천): 서브도메인 방식 (`erp.ledmega.cloud` 로 접속 시 etErp 연결)**
기존 HTTP 80 라우팅이 `*.ledmega.cloud`를 매핑해주고 있으므로 아주 쉽게 셋업할 수 있습니다.
```nginx
# ── 4. HTTPS(443): etERP 프록시 (서브도메인: erp.ledmega.cloud) ─────────────────
server {
    listen 443 ssl;
    server_name erp.ledmega.cloud;

    ssl_certificate     /etc/letsencrypt/live/ledmega.cloud/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/ledmega.cloud/privkey.pem;

    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;
    client_max_body_size 50m;

    proxy_set_header Host              $http_host;
    proxy_set_header X-Real-IP         $remote_addr;
    proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto https;
    proxy_http_version 1.1;
    proxy_set_header Upgrade    $http_upgrade;
    proxy_set_header Connection $connection_upgrade;

    location / {
        proxy_pass http://172.17.0.1:8081;  # etERP (8081) 로 연결
    }
}
```

**👉 방법 B: 기존 서버와 동일한 도메인 + 새로운 외부 포트 (예: 18792)**
만약 서브도메인이 싫고 그냥 포트로만 구분하고 싶다면 아래처럼 등록합니다.
```nginx
# ── 4. HTTPS(18792): etERP 프록시 (ledmega.cloud:18792) ─────────────────────
server {
    listen 18792 ssl;
    server_name ledmega.cloud;

    ssl_certificate     /etc/letsencrypt/live/ledmega.cloud/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/ledmega.cloud/privkey.pem;

    proxy_set_header Host              $http_host;
    proxy_set_header X-Forwarded-Proto https;
    proxy_http_version 1.1;
    proxy_set_header Upgrade    $http_upgrade;
    proxy_set_header Connection $connection_upgrade;

    location / {
        proxy_pass http://172.17.0.1:8081; # etERP (8081) 로 연결
    }
}
```
방법 A, B 중 원하시는 것을 선택하셔서, 운영 중이신 우분투 Nginx 설정 하단에 붙여넣고 Nginx Docker 컨테이너를 재시작(`docker restart nginx`) 해주시면 됩니다!

     ```nginx
     server {
     ```

## Q25. 가비아 도메인 + Cloudflare + Let's Encrypt(Nginx) 연동 가이드

| 항목 | 내용 |
|------|------|
| 🙋 **질문일시** | 2026-03-30 15:12 |

> (이전 Nginx 설정: ssl_certificate /etc/letsencrypt/live/ledmega.cloud/fullchain.pem;)

**답변:**

정확히 파악했습니다! 현재 회원님의 아키텍처는 **Cloudflare (프록시/디도스방어) ➜ iptime 공유기 ➜ Ubuntu Nginx (Let's Encrypt 정식 인증서)** 로 연결되는 아주 훌륭한 "Full (strict) SSL" 이중 보안 구조입니다.

새로 구입하신 `ectech.cloud`를 이 구조와 **완벽히 동일하게** 세팅하시려면 다음 4단계를 거치시면 됩니다.

**1단계: 가비아에서 네임서버 변경**
- 가비아 도메인 관리 페이지로 이동하여 `ectech.cloud`의 **네임서버(Name Server)** 를 Cloudflare에서 제공하는 네임서버(예: `hera.ns.cloudflare.com` 등)로 변경합니다. (반영에 1~2시간 소요될 수 있습니다.)

**2단계: Cloudflare 설정 (A 레코드 추가)**
- Cloudflare에 로그인하여 `ectech.cloud` 사이트를 새로 추가합니다.
- DNS 메뉴에서 **`A 레코드`** 를 추가하고, **집(iptime)의 공인 IP** 를 입력합니다.
- **주황색 구름 모양(프록시 상태)** 을 켜두어 Cloudflare SSL을 활성화합니다.
- SSL/TLS 모드는 호환성을 위해 **"전체(Full)"** 모드로 유지합니다.

**3단계: Ubuntu 서버에 Let's Encrypt 인증서 발급 (중요)**
도메인이 달라졌으므로 기존 `ledmega.cloud` 인증서를 쪼개 쓸 수는 없습니다. **`ectech.cloud`용 인증서를 새로 발급**받아야 합니다.
이전에 하셨던 방식과 동일하게 Nginx 컨테이너나 우분투 호스트에서 `certbot`을 실행하여 인증서를 발급받습니다.
```bash
# 예시 (Nginx 잠깐 내리고 standalone으로 받거나 webroot 방식 사용)
sudo certbot certonly --standalone -d ectech.cloud -d www.ectech.cloud
```
성공하면 아까 보여주신 설정처럼 `/etc/letsencrypt/live/ectech.cloud/fullchain.pem` 이 생성됩니다.

**4단계: Nginx `default.conf` 수정**
이제 Nginx 설정 하단에 방금 발급받은 인증서 경로를 넣은 새 블록을 추가해주면 끝납니다!
```nginx
server {
    listen 443 ssl;
    server_name ectech.cloud www.ectech.cloud;

    # 3단계에서 새로 발급받은 ectech.cloud 의 인증서 경로
    ssl_certificate     /etc/letsencrypt/live/ectech.cloud/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/ectech.cloud/privkey.pem;

    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;
    client_max_body_size 50m;

    location / {
        proxy_pass http://172.17.0.1:8081;  # etERP(Spring Boot)로 연결
    }
}
```

이 흐름대로 세팅하시면 기존 `ledmega.cloud`가 돌아가는 **완벽히 똑같은 원리와 포트(443) 위에서**, 도메인 이름만 변경된 상태로 사내 ERP에 접속하실 수 있게 됩니다! (해당 내용도 Q25번으로 QA 로그에 기록했습니다.)

   - **Proxy status (프록시 상태)**: 주황색 구름 모양(Proxied)이 켜져 있는지 확인

리눅스 서버에서 `./gradlew` 명령어로 빌드하려면 프로젝트 안에 실행용 파일들이 포함되어 있어야 하는데, 이 파일들이 누락되어 발생한 문제입니다. **Anti Gravity(AI)**인 제가 직접 윈도우 PC의 환경을 진단하고 수동으로 파일을 만들어 GitHub에 올렸습니다.

**조치 내용:**
1.  **Gradle 래퍼 파일 다운로드**: `gradle-wrapper.properties`, `gradlew` (리눅스용), `gradlew.bat` (윈도우용) 파일을 Gradle 공식 저장소(v8.12.1)에서 직접 내려받아 배치했습니다.
2.  **바이너리(`jar`) 확보**: 실행 핵심 파일인 `gradle-wrapper.jar`를 수동으로 확보하여 `gradle/wrapper/` 경로에 배치했습니다.
3.  **Git 업로드**: `.gitignore` 규칙에 의해 무시될 수 있는 `.jar` 파일 등을 **강제(`git add -f`)**로 깃허브에 Push 완료했습니다.

**우분투 서버에서 이제 하실 일:**
1.  `git pull origin main` (최신 래퍼 파일 다운로드)
2.  `chmod +x gradlew` (실행기 권한 부여)
3.  `./run.sh rebuild` (다시 빌드 시작!)

이제 우분투 서버가 스스로 빌드 엔진을 돌려 서버를 띄울 준비가 모두 끝났습니다!
