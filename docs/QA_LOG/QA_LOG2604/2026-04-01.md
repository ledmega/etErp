## Q2. Nginx 504 Gateway Time-out 에러 발생 원인과 해결 방법

| 항목 | 내용 |
|------|------|
| 🙋 **질문일시** | 2026-04-01 11:27 |
| 🤖 **답변일시** | 2026-04-01 11:32 |

**질문:**
> 왜 접속이 안될까?
> (첨부 이미지: ectech.ledmega.cloud 접속 시 Nginx 504 Gateway Time-out 에러, Cloudflare DNS 설정 화면)

**답변:**
보내주신 이미지를 보면 **Cloudflare DNS 설정 (A 레코드, CNAME 등)은 정상적으로 Nginx 서버(`218.237.70.67`)까지 연결**을 해주고 있습니다. 웹 브라우저가 Nginx 서버와 통신은 성공했기 때문에 Nginx에서 뱉어내는 504 에러 페이지가 보이는 것입니다.

하지만 Nginx가 받은 요청을 뒷단(Backend)에 있는 **Spring Boot 서버(주로 포트 8080)로 전달(Proxy)하려고 했지만, 응답을 받지 못해 시간 초과(Time-out)**가 발생한 상황입니다. 

주요 원인과 해결을 위한 점검 체크리스트는 다음과 같습니다.

### 1. Spring Boot 서버 (Backend) 구동 상태 확인
가장 흔한 원인은 뒷단의 Spring Boot 애플리케이션이 죽어있거나 완전히 구동되지 않은 경우입니다.
* 우분투 서버에서 프로세스가 떠있는지 확인:
  ```bash
  ps -ef | grep java
  ```
* 포트(예: 8080)가 정상적으로 열려있는지 확인:
  ```bash
  netstat -ntlp | grep 8080
  ```
* 서버 에러 로그(`nohup.out` 또는 스프링 로그 파일)를 확인하여 DB 연결 실패 등의 이유로 서버가 중간에 멈췄거나 오류를 뿜고 있는지 점검하세요.

### 2. Nginx proxy_pass 설정 확인
Nginx 설정 파일(보통 `/etc/nginx/sites-available/...` 또는 `/etc/nginx/nginx.conf`)에서 `proxy_pass`가 가리키는 주소와 포트가 맞는지 확인합니다.
* 예: `proxy_pass http://localhost:8080;` 또는 `http://127.0.0.1:8080;`
* 만약 백엔드를 Docker 컨테이너로 구동했다면 서로 네트워크가 격리되어 Nginx가 localhost로 찾지 못할 수 있습니다.

### 3. 방화벽 (UFW / iptables) 문제
Nginx에서 백엔드 앱으로 통신할 때 내부 방화벽에서 패킷을 DROP하여 타임아웃이 발생할 수 있습니다. (Reject면 502 Bad Gateway가 뜹니다)
* UFW(우분투 방화벽) 상태 확인:
  ```bash
  sudo ufw status
  ```
* 필요하다면 내부 로컬 통신이나 8080 포트가 차단되어 있는지 확인해야 합니다.

**✅ 가장 먼저 해보실 조치:**
서버(218.237.70.67)에 SSH로 접속하셔서 Spring Boot 서버의 **에러 로그를 확인**하고, `curl http://localhost:8080` 명령어로 백엔드 서버가 로컬에서 정상적으로 응답하는지 테스트해 보시기 바랍니다!

## Q3. ufw status 및 curl 테스트 결과 분석 (504 에러 원인 추적)

| 항목 | 내용 |
|------|------|
| 🙋 **질문일시** | 2026-04-01 12:13 |
| 🤖 **답변일시** | 2026-04-01 12:18 |

**질문:**
> (첨부 이미지: `sudo ufw status` 결과와 `curl -v http://localhost:8080` 결과 캡처 화면)

**답변:**
보내주신 스크린샷 2장을 분석해 보면 매우 중요한 단서들을 확인할 수 있습니다.

### 📌 캡처 화면 분석 결과
1. **Spring Boot (Backend)는 정상 구동 중입니다.**
   * `curl` 결과를 보면 `localhost:8080`으로 아주 잘 연결되었고, 응답으로 `303 See Other (Location: /dashboard)` 정상 응답(리다이렉트)을 내려주고 있습니다. 서버 애플리케이션 문제는 **아닙니다.**
2. **방화벽(UFW) 설정에 제한이 걸려 있습니다.**
   * `8080/tcp` 포트의 ALLOW(허용) 규칙이 `Anywhere`가 아니라 **`172.17.0.0/16`** (Docker의 기본 네트워크 대역)으로 설정되어 있습니다.

### 💡 504 Gateway Time-out 의 진짜 원인 2가지 가능성

Backend 서버가 정상임에도 Nginx에서 연결하지 못해 타임아웃이 나는 원인은 다음 2가지 중 하나일 확률이 99%입니다.

**원인 1. Nginx(Host 설치)의 proxy_pass 대상 주소 오류**
만약 Nginx가 Docker가 아닌 우분투 호스트에 직접 설치된 상태라면, `/etc/nginx/sites-available/...` (또는 nginx.conf) 설정 파일 안의 주소를 확인해야 합니다.
`proxy_pass`가 호스트의 공인IP(`http://218.237.70.67:8080` 등)로 되어 있다면, 현재 UFW가 8080 포트는 `172.17.0.0/16`만 허용하고 외부 접근을 막았기 때문에 통신이 차단되어 타임아웃이 발생합니다.
* **👉 해결 방법**: Nginx 설정의 `proxy_pass` 대상을 **`http://127.0.0.1:8080;`** 또는 **`http://localhost:8080;`** 으로 변경하고 Nginx를 재시작(`sudo systemctl restart nginx`) 하세요.

**원인 2. Nginx를 Docker 컨테이너로 구동한 상태**
Nginx가 Docker로 떠있고 Spring Boot는 호스트에 떠있는 상황이라면, Nginx 컨테이너 안에서 `localhost`를 호출하면 Spring Boot가 아닌 Nginx 컨테이너 자기 자신을 가리키게 되어 연결에 실패합니다.
* **👉 해결 방법**: Nginx 설정의 `proxy_pass` 대상을 Docker 호스트 IP인 **`http://172.17.0.1:8080;`** 로 변경해 주어야 합니다. (다행히 호스트 UFW에는 `172.17.0.0/16` 대역의 접속을 허용해 두었으므로 이 접근은 통과됩니다.)

**✅ 다음 확인 사항:**
우분투 서버에서 Nginx 설정 파일 (`sudo nano /etc/nginx/sites-enabled/default` 또는 해당 설정파일)을 열어 `proxy_pass` 항목이 정확히 어떻게 적혀 있는지 확인하고, 위 원인 1 또는 2에 맞게 수정해 보시기 바랍니다!

## Q4. Nginx 설정 분석 완료 (504 원인: 포트 불일치 및 방화벽)

| 항목 | 내용 |
|------|------|
| 🙋 **질문일시** | 2026-04-01 12:16 |
| 🤖 **답변일시** | 2026-04-01 12:20 |

**질문:**
> 현재 nginx는 도커로 기동되어 있고 3개의 프로젝트(mega, openclaw, etErp)가 떠있음.
> Nginx `default.conf` 파일 내용 첨부:
> - `mega`: proxy_pass http://172.17.0.1:8080;
> - `openclaw`: proxy_pass http://172.17.0.1:18789;
> - `etErp`: proxy_pass http://172.17.0.1:8081;

**답변:**
원인을 명확히 찾았습니다! 설정 파일과 앞서 보여주신 `curl` 테스트, 그리고 방화벽 상태에서 모순점이 발생하여 504 Timeout이 일어나고 있었습니다.

### 💡 문제의 원인 2가지 (포트 불일치)

1. **etErp 서버 구동 포트 불일치**
   * Nginx 설정 파일에서는 `ectech.ledmega.cloud` (etErp) 요청을 **`8081` 포트**로 보내도록 설정(`proxy_pass http://172.17.0.1:8081;`)하셨습니다.
   * 그러나 직전에 `etErp` 디렉토리에서 실행하신 `curl http://localhost:8080` 테스트 결과를 보면, 현재 **etErp 애플리케이션은 8081 포트가 아니라 8080 포트로 띄워져 있습니다.** (application.yml 설정이 8080으로 되어 있을 것입니다.)
   * **결과:** Nginx는 8081 포트를 찌르는데 거기엔 아무것도 없으니 타임아웃이 발생한 것입니다. 현재 `mega`와 `etErp`가 모두 `8080` 포트를 점유하려고 충돌하고 있을 수 있습니다.

2. **방화벽(UFW)에 8081 포트 미개방**
   * Nginx(Docker)가 호스트(172.17.0.1)의 `8081` 포트에 접근해야 하는데, 앞의 UFW 상태를 보면 `8080/tcp`만 `172.17.0.0/16`에 열려 있고 **`8081/tcp`는 아예 열려있지 않습니다.**

### 🛠️ 해결 방법 (Action Plan)

이 문제를 해결하려면 다음 **3단계**를 진행해 주시면 됩니다!

**1단계: 방화벽에 8081 포트 허용 (Docker 내부 통신용)**
우분투 터미널에서 다음 명령어를 실행하여 8081 포트도 Nginx가 접근할 수 있게 열어줍니다.
```bash
sudo ufw allow from 172.17.0.0/16 to any port 8081
sudo ufw status # 8081이 추가되었는지 확인
```

**2단계: etErp 프로젝트의 포트를 8081로 변경**
etErp 소스코드의 설정 파일 (아마도 `application-dev.yml` 또는 `application.yml`)을 열어 포트를 변경해 줍니다.
```yaml
server:
  port: 8081
```

**3단계: etErp 다시 빌드 후 재기동**
etErp 서버를 중지한 뒤 8081 포트로 다시 띄우시고, 정상적으로 8081 포트가 떴는지 로컬에서 테스트해 봅니다.
```bash
# 로컬 통신 테스트 (이번엔 8081로)
curl -v http://localhost:8081
```

이후 웹 브라우저에서 `https://ectech.ledmega.cloud` 로 접속해 보시면 504 오류 없이 정상적으로 화면이 뜰 것입니다!

## Q5. etErp 애플리케이션의 포트를 8089로 변경하는 방안

| 🙋 **질문일시** | 2026-04-01 12:18 |
| 🤖 **답변일시** | 2026-04-01 12:21 |

**질문:**
> 8089 로 변경할까?

**답변:**
네, `8089`로 변경하셔도 좋습니다! 다만, 방금 올려주신 Nginx `default.conf` 설정 파일에 etErp의 대상 포트가 `8081`로 하드코딩 되어 있으므로, **포트를 `8089`로 정하셨다면 총 3군데를 모두 `8089`로 맞추어 주셔야 합니다.**

1. **Spring Boot (`application.yml` 또는 `application-dev.yml`):**
   ```yaml
   server:
     port: 8089
   ```

2. **Nginx 설정 (`default.conf`):**
   ```nginx
   # ── 5. HTTPS(443): etERP 서브도메인 프록시 ──
   location / {
       proxy_pass http://172.17.0.1:8089;  # <--- 이 부분을 8089로 수정
   }
   ```
   수정 후에는 Nginx 컨테이너를 꼭 다시 로드해야 합니다.
   `docker restart Nginx컨테이너이름` 또는 `docker exec Nginx컨테이너이름 nginx -s reload`

3. **우분투 방화벽 (UFW):**
   도커(Nginx)가 호스트의 `8089` 포트에 연결할 수 있도록 문을 열어줍니다.
   ```bash
   sudo ufw allow from 172.17.0.0/16 to any port 8089
   ```

> 💡 **추천 사항:** 
> 만약 Nginx 설정을 다시 수정하고 재시작(reload)하는 것이 번거로우시다면, 애플리케이션 포트를 Nginx 설정에 이미 적혀있는 **`8081`** 로 맞추는 것이 작업량을 제일 줄일 수 구멍이긴 합니다!

## Q8. Docker 구동 중인 Nginx 컨테이너 재시작 방법

| 항목 | 내용 |
|------|------|
| 🙋 **질문일시** | 2026-04-01 12:29 |
| 🤖 **답변일시** | 2026-04-01 12:31 |

**질문:**
> (docker ps 화면 첨부, Nginx 컨테이너 이름: `nginx-proxy`)
> nginx 재시작 어떻게 하는거지?

**답변:**
보여주신 `docker ps` 결과를 보니 Nginx 컨테이너의 이름이 **`nginx-proxy`** 로 되어 있습니다.

Nginx 설정(`default.conf`)을 수정한 후 이를 Nginx에 반영하려면 우분투 서버 터미널에서 아래 두 가지 명령어 중 하나를 입력하시면 됩니다.

**방법 1: 설정만 안전하게 새로고침 (가장 추천)**
서비스 중단 없이 변경된 설정 파일만 다시 읽어들입니다. 통신이 순간적으로 끊기지 않습니다.
```bash
docker exec nginx-proxy nginx -s reload
```

**방법 2: 컨테이너 완전 재시작**

## Q9. 접속 시 브라우저 기본 로그인 창(HTTP Basic Auth)이 뜨는 이유

| 항목 | 내용 |
|------|------|
| 🙋 **질문일시** | 2026-04-01 12:32 |
| 🤖 **답변일시** | 2026-04-01 12:35 |

**질문:**
> (브라우저에서 `https://ectech.ledmega.cloud` 접속 시 팝업 형태의 '로그인' 창이 나타난 캡처 화면)
> 이거 왜이런거야? 

**답변:**
🎉 **먼저 연결 성공을 축하드립니다!**
504 에러가 사라지고 저 창이 떴다는 것은 드디어 Nginx 방화벽과 포트 문제가 완벽히 해결되어 **Spring Boot (etErp) 백엔드 생살에 정상적으로 닿았다는 확실한 증거**입니다.

이렇게 팝업창이 뜨는 현상은 Nginx나 도커의 오류가 아니라, **Spring Security** 때문입니다.

### 💡 원인 설명
스프링 부트 프로젝트 `build.gradle.kts`에 `spring-boot-starter-security` 라이브러리가 존재하면, **기본적으로 '모든 요청을 비인가 접근으로 간주하고 잠궈버리는 것'**이 스프링 시큐리티의 기본 정책입니다.
우리가 아직 로그인 화면(HTML)이나 보안 우회 설정(`SecurityConfig`)을 명시적으로 연결해주지 않았기 때문에, 프레임워크가 강제로 브라우저 내장 로그인 창(**HTTP Basic Authentication**)을 띄우는 것입니다.

### 🛠️ 해결 방안 / 당장 확인하는 법

**대처 1: 기본 계정으로 로그인해보기**
스프링 부트 서버 구동 시 터미널(혹은 nohup 로그)에 아래와 비슷한 비밀번호가 무작위로 생성되어 찍혀있을 것입니다.
`Using generated security password: 9f0322d4-xxxx-xxxx...`
* **사용자 이름**: `user`
* **비밀번호**: 로그에 찍힌 비밀번호
입력하시면 통과되어 서버의 리턴값이나 페이지를 보실 수 있습니다.

**대처 2: SecurityConfig 클래스 설정 추가 (가장 확실함)**
지금은 개발 초기 단계이거나 커스텀 폼 로그인을 만드실 계획일 테니, 이 기본 팝업을 끄고 원하는 URL은 통과시키도록 소스코드를 작성해야 합니다.
```java
// 예시: SecurityConfig 설정에서 Basic Auth 비활성화
@Bean
public SecurityWebFilterChain springSecurityFilterChain(ServerHttpSecurity http) {
    return http
            .httpBasic(basic -> basic.disable()) // 기본 팝업창 끄기
            .formLogin(form -> form.disable())
            .authorizeExchange(exchanges -> exchanges
                    .pathMatchers("/").permitAll() // 메인은 누구나 접근
                    .anyExchange().authenticated() // 나머지는 인증 필요
            )
            // ... (기타 JWT 필터 설정 등)
            .build();
}
```

이 현상은 아주 지극히 정상이므로 걱정하지 않으셔도 되며, 팝업이 성가시다면 `SecurityConfig` 파일에서 `httpBasic.disable()` 처리를 해주시면 금방 해결됩니다!

## Q12. 메인 접속 시 서버 내부 오류(E199) JSON 응답이 나타나는 이유

| 🙋 **질문일시** | 2026-04-01 12:51 |

> (브라우저에서 접속 시 `{"success": false, "code": "E199", "message": "서버 내부 오류가 발생했습니다."}` 화면 출력)
> 이거 왜이런거야?

**답변:**
이 화면은 **완벽한 "서비스 구동 완료"의 100점짜리 인증샷**입니다! 🎉

### 💡 왜 에러 메시지가 예쁘게 뜨는 건가요?
1. Nginx를 통과하여 스프링 부트에 도달한 브라우저가 첫 화면인 `/` 경로를 달라고 요청했습니다.
2. 하지만 아직 저희가 **스프링 부트 소스 안에 첫 번째 화면(`.html`) 이나 컨트롤러 로직을 하나도 만들지 않았습니다.** 그래서 "화면이 없다(404 Not Found)"는 예외가 스프링 내부에서 터졌습니다.
3. 이를 우리가 프로젝트 초기에 공통으로 만들어둔 **`GlobalExceptionHandler`** 가 친절하게 낚아채서, 정해진 `ApiResponse` 규격(JSON 형태)으로 깔끔하게 포장해 브라우저에 던져준 것입니다.

정리하면, 에러가 발생한 게 맞지만 **통제가 완벽하게 되는 우리가 설계한 에러 응답 시스템이 잘 동작하고 있다**는 아주 훌륭한 결과입니다!

### 🛠️ 대처 방안 (메인 화면 만들기)
더 이상 에러 메시지가 뜨지 않고 정상 페이지가 뜨도록, 테스트 겸 간단한 `RootController.java`를 하나 생성해 두었습니다.

* 생성 파일: `src/main/java/com/et/erp/common/controller/RootController.java`
* 하는 일: 경로 `/` 로 요청이 오면 "Welcome to etERP API Server!" 라는 환영 메시지를 반환합니다.

```java
@RestController
public class RootController {
    @GetMapping("/")
    public Mono<Map<String, String>> welcome() {
         return Mono.just(Map.of(
            "message", "Welcome to etERP API Server!",
            "status", "Running",
            "version", "1.0.0"
        ));
    }
}
```

이제 다시 서버에서 커밋 내역을 받아(`git pull`) 한 번만 더 재구동해 보시면, 에러 대신 성공 메시지가 응답되는 것을 예쁘게 확인하실 수 있을 것입니다! (방금 이 내용과 소스 코드 모두 Github에 `push` 해두었습니다!)

이제 `sys_user` 테이블과 직접 교신하며 회원 생성 및 패스워드 검증을 처리합니다.

* `com/et/erp/auth/controller/AuthApiController.java` : 데이터(JSON) 응답용 웹 API
