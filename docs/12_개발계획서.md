## 1. 프로젝트 개요

### 1-1. 배경 및 목적

현재 사내 업무 처리가 Excel, 개별 시스템, 수작업 등으로 분산되어 있어
데이터 일관성 확보 및 업무 효율화를 위한 통합 ERP 시스템을 구축합니다.

본 프로젝트는 **실무 ERP 개발**과 더불어 **Java Reactive Programming(Spring WebFlux)**
학습을 병행하는 것을 목표로 합니다.

### 1-2. 프로젝트 목표

| 구분 | 목표 |
|------|------|
| **비즈니스 목표** | 전사 업무 통합 및 데이터 일원화 |
| **기술 목표** | Spring WebFlux / R2DBC 기반 Reactive 시스템 구축 |
| **학습 목표** | Java Reactive Programming 실전 적용 능력 확보 |

### 1-3. 프로젝트 정보

| 항목 | 내용 |
|------|------|
| **시스템명** | etERP |

## 5. 개발 방법론 및 규칙

### 5-1. 개발 방식
- **기능 단위 개발**: 모듈별로 Controller → Service → Repository → Entity → DTO 순서로 개발
- **테스트 병행**: 주요 Service 메서드는 `StepVerifier` 기반 단위 테스트 작성
- **DB 마이그레이션**: Flyway로 스키마 변경 이력 관리 (`docs/sql/` 파일 선행 작성)
- **문서 선행 작성**: 기능 개발 전 API 설계 문서 먼저 업데이트

### 5-2. Git 브랜치 전략

```
main          ← 운영 배포용 (PR만 허용)
  └─ develop  ← 개발 통합 브랜치
       ├─ feat/auth              ← 인증 기능
       ├─ feat/hr-employee       ← 직원 관리
       ├─ feat/accounting-voucher ← 전표 관리
       └─ fix/login-bug          ← 버그 수정
```

### 5-3. 커밋 메시지 규칙

```
feat:     새로운 기능 추가      (feat: 직원 등록 API 구현)
fix:      버그 수정             (fix: 로그인 토큰 만료 오류 수정)
docs:     문서 수정             (docs: API 설계서 업데이트)
refactor: 리팩토링              (refactor: EmployeeService 로직 분리)
test:     테스트 코드 추가       (test: EmployeeService 단위 테스트 추가)
chore:    빌드/설정 변경        (chore: build.gradle 의존성 업데이트)
sql:      DB 스크립트 변경      (sql: hr_leave 테이블 컬럼 추가)
```

### 5-4. 코딩 원칙
- `block()` 절대 사용 금지 (Reactive 원칙)
- Entity에 `@Setter` 금지 → 비즈니스 메서드로 상태 변경
- 모든 public 메서드 한글 Javadoc 작성 (학습 목적)
- 로그에 개인정보(비밀번호, 계좌번호 등) 출력 금지
