## 1. Reactive Programming 이란?

### 전통적인 방식 (Blocking / Imperative)
```
요청 → 스레드 점유 → DB 쿼리 대기(블로킹) → 응답 반환 → 스레드 해제
```
- 요청 1건당 스레드 1개 점유
- DB 응답 올 때까지 **스레드가 멈춰서 기다림** (낭비)
- 동시 요청이 많으면 → 스레드 부족 → 성능 저하

### 리액티브 방식 (Non-Blocking / Reactive)
```
요청 → 작업 등록 → 스레드 반환 → (DB 완료 이벤트) → 결과 처리 → 응답
```
- 스레드가 DB를 기다리지 않고 **다른 작업을 처리**
- 적은 스레드로 많은 동시 요청 처리 가능
- **이벤트 기반** 비동기 처리

## 2. 핵심 개념: Mono & Flux

### Mono<T> — 0개 또는 1개의 데이터
```java
// Mono: 단일 결과 (조회 1건, 등록, 수정, 삭제에 사용)
Mono<Employee> mono = employeeRepository.findById(1L);

// 비어있는 Mono (결과 없음)
Mono<Void> empty = Mono.empty();

// 즉시 값 발행
Mono<String> just = Mono.just("Hello");

// 에러 Mono
Mono<Employee> error = Mono.error(new RuntimeException("오류 발생"));
```

### Flux<T> — 0개 이상의 데이터 스트림
```java
// Flux: 복수 결과 (목록 조회에 사용)
Flux<Employee> flux = employeeRepository.findAll();

// 즉시 여러 값 발행
Flux<Integer> numbers = Flux.just(1, 2, 3, 4, 5);

// 리스트에서 Flux 생성
Flux<String> fromList = Flux.fromIterable(List.of("A", "B", "C"));
```

### 핵심 차이점
```
Mono<Employee>  = Optional<Employee>의 비동기 버전 (최대 1개)
Flux<Employee>  = List<Employee>의 비동기 버전 (0~N개)
```

## 3. 기술 스택 변경사항 (기존 MVC → WebFlux)

| 항목 | Spring MVC (기존) | Spring WebFlux (변경) |
|------|-------------------|-----------------------|
| **웹 프레임워크** | Spring MVC | Spring WebFlux |
| **서버** | Embedded Tomcat | Embedded Netty (논블로킹) |
| **DB 접근** | Spring Data JPA (블로킹) | Spring Data R2DBC (논블로킹) |
| **DB 드라이버** | JDBC | R2DBC |
| **반환 타입** | `Employee`, `List<Employee>` | `Mono<Employee>`, `Flux<Employee>` |
| **보안** | Spring Security (Servlet) | Spring Security Reactive |
| **트랜잭션** | `@Transactional` | `@Transactional` (Reactive 지원) |
| **쿼리** | QueryDSL / JPQL | R2DBC + SQL 직접 또는 R2DBC Repository |

## 7. Operator 주요 메서드 학습

### map() — 데이터 변환 (동기)
```java
// Employee Entity → EmployeeResponse DTO 변환
Mono<EmployeeResponse> response = employeeRepository.findById(id)
    .map(employee -> EmployeeResponse.from(employee));
    // map: 1:1 동기 변환 (내부에서 Mono/Flux를 반환하면 안됨)
```

### flatMap() — 데이터 변환 (비동기)
```java
// 직원 조회 후 부서 정보도 조회 (비동기 연쇄)
Mono<EmployeeDetailResponse> detail = employeeRepository.findById(id)
    .flatMap(employee ->
        // flatMap: 내부에서 Mono를 반환할 때 사용 (비동기 → 비동기)
        deptRepository.findById(employee.getDeptId())
            .map(dept -> EmployeeDetailResponse.of(employee, dept))
    );
```

### filter() — 조건 필터링
```java
// 재직 중인 직원만 필터
Flux<Employee> active = employeeRepository.findAll()
    .filter(emp -> "A".equals(emp.getEmpStatus()));
    // filter: 조건 불만족 시 해당 항목 제외
```

### switchIfEmpty() — 비어있을 때 대체
```java
// 직원 없을 때 예외 발생
Mono<Employee> employee = employeeRepository.findById(id)
    .switchIfEmpty(
        // switchIfEmpty: Mono.empty()일 때 대체 Mono 실행
        Mono.error(new ErpException(ErrorCode.EMPLOYEE_NOT_FOUND))
    );
```

### doOnNext() — 사이드 이펙트 (로그 등)
```java
Mono<Employee> employee = employeeRepository.findById(id)
    .doOnNext(emp ->
        // doOnNext: 데이터를 소비하지 않고 로그/알림 등 부수 효과만 실행
        log.info("[직원조회] empId={}, name={}", emp.getEmpId(), emp.getEmpName())
    );
```

### zipWith() — 두 Mono 병렬 실행 후 결합
```java
// 직원정보와 근태정보를 병렬로 조회 후 합치기
Mono<EmployeeDetailResponse> result =
    employeeRepository.findById(empId)
        .zipWith(attendanceRepository.findTodayByEmpId(empId))
        // zipWith: 두 Mono를 동시에 실행하고 둘 다 완료되면 결합
        .map(tuple -> {
            Employee emp = tuple.getT1();     // 첫 번째 결과
            Attendance att = tuple.getT2();   // 두 번째 결과
            return EmployeeDetailResponse.of(emp, att);
        });
```

### collectList() — Flux → Mono<List>
```java
// Flux<Employee>를 List로 모아서 처리
Mono<List<Employee>> list = employeeRepository.findAll()
    .collectList();
    // collectList: 모든 스트림 항목을 List로 수집
```

### onErrorResume() — 에러 처리
```java
Mono<Employee> safe = employeeRepository.findById(id)
    .onErrorResume(ex -> {
        // onErrorResume: 에러 발생 시 대체 Mono 반환
        log.error("[직원조회 오류] id={}, error={}", id, ex.getMessage());
        return Mono.error(new ErpException(ErrorCode.EMPLOYEE_NOT_FOUND));
    });
```

## 8. 리액티브 Controller 작성

```java
/**
 * 직원 관리 컨트롤러 (Reactive)
 *
 * - 모든 메서드는 Mono 또는 Flux를 반환합니다.
 * - block(), blockFirst() 등 블로킹 메서드는 절대 사용하지 마세요!
 *   (리액티브의 장점을 완전히 무너뜨립니다)
 */
@RestController
@RequestMapping("/api/v1/hr/employees")
@RequiredArgsConstructor
@Slf4j
public class EmployeeController {

    private final EmployeeService employeeService;

    /**
     * 직원 목록 조회
     * - Flux<EmployeeResponse> 반환: 여러 건 스트리밍
     */
    @GetMapping
    public Flux<EmployeeResponse> findAll() {
        log.info("[직원목록 조회] 요청");
        return employeeService.findAll();
    }

    /**
     * 직원 단건 조회
     * - Mono<EmployeeResponse> 반환: 단일 건
     */
    @GetMapping("/{empId}")
    public Mono<EmployeeResponse> findById(@PathVariable Long empId) {
        log.info("[직원상세 조회] empId={}", empId);
        return employeeService.findById(empId);
    }

    /**
     * 직원 등록
     * - 요청 바디를 Mono<EmployeeRequest>로 받음
     * - @Valid 검증 포함
     */
    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public Mono<EmployeeResponse> create(
            @Valid @RequestBody Mono<EmployeeRequest> requestMono) {
        // requestMono.flatMap: 요청 DTO가 도착하면 서비스 호출
        return requestMono.flatMap(employeeService::create);
    }

    /**
     * 직원 수정
     */
    @PutMapping("/{empId}")
    public Mono<EmployeeResponse> update(
            @PathVariable Long empId,
            @Valid @RequestBody Mono<EmployeeRequest> requestMono) {
        return requestMono.flatMap(req -> employeeService.update(empId, req));
    }

    /**
     * 직원 삭제 (논리 삭제)
     * - Mono<Void> 반환: 응답 본문 없음
     */
    @DeleteMapping("/{empId}")
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public Mono<Void> delete(@PathVariable Long empId) {
        return employeeService.delete(empId);
    }
}
```

## 9. 리액티브 Service 작성

```java
/**
 * 직원 서비스 구현체 (Reactive)
 *
 * 핵심 원칙:
 * 1. 절대 block() 사용 금지
 * 2. Operator 체이닝으로 비즈니스 로직 구성
 * 3. 에러는 Mono.error()로 전파
 * 4. 트랜잭션은 @Transactional 사용 (Reactive 지원됨)
 */
@Service
@RequiredArgsConstructor
@Slf4j
public class EmployeeServiceImpl implements EmployeeService {

    private final EmployeeRepository employeeRepository;

    /**
     * 전체 직원 조회
     * - Flux로 스트리밍 반환 (데이터가 생길 때마다 즉시 전달)
     */
    @Override
    public Flux<EmployeeResponse> findAll() {
        return employeeRepository.findAllByDeletedYn("N")
            // map: Entity → Response DTO 변환 (동기 변환이므로 map 사용)
            .map(EmployeeResponse::from)
            // doOnNext: 각 항목을 처리할 때 로그 기록 (데이터 변환 없음)
            .doOnNext(emp -> log.debug("[직원목록] empId={}", emp.getEmpId()));
    }

    /**
     * 직원 단건 조회
     * - switchIfEmpty로 없을 경우 404 예외 처리
     */
    @Override
    public Mono<EmployeeResponse> findById(Long empId) {
        return employeeRepository.findById(empId)
            // switchIfEmpty: 조회 결과가 없으면(Mono.empty) 예외 발생
            .switchIfEmpty(Mono.error(
                new ErpException(ErrorCode.EMPLOYEE_NOT_FOUND)
            ))
            // map: 정상 조회 시 DTO 변환
            .map(EmployeeResponse::from);
    }

    /**
     * 직원 등록
     * - @Transactional: Reactive 트랜잭션 관리
     */
    @Override
    @Transactional
    public Mono<EmployeeResponse> create(EmployeeRequest request) {
        // 1단계: 사번 중복 체크
        return employeeRepository.existsByEmpNo(request.getEmpNo())
            // flatMap: 중복 체크 결과(Boolean)에 따라 다음 Mono 결정
            .flatMap(exists -> {
                if (exists) {
                    // 중복이면 에러 Mono 반환
                    return Mono.error(new ErpException(ErrorCode.DUPLICATE_EMP_NO));
                }
                // 2단계: Entity 생성 후 저장
                Employee employee = Employee.create(request);
                return employeeRepository.save(employee);
            })
            // map: 저장된 Entity → Response DTO
            .map(EmployeeResponse::from)
            // doOnSuccess: 저장 성공 시 로그
            .doOnSuccess(res ->
                log.info("[직원등록 완료] empId={}, empNo={}", res.getEmpId(), res.getEmpNo())
            );
    }

    /**
     * 직원 삭제 (논리 삭제)
     */
    @Override
    @Transactional
    public Mono<Void> delete(Long empId) {
        return employeeRepository.findById(empId)
            .switchIfEmpty(Mono.error(new ErpException(ErrorCode.EMPLOYEE_NOT_FOUND)))
            // flatMap: 조회된 Entity를 수정 후 저장
            .flatMap(employee -> {
                employee.softDelete(); // deleted_yn = 'Y' 처리
                return employeeRepository.save(employee);
            })
            // then: 이전 결과를 버리고 Mono<Void> 반환
            .then();
    }
}
```

## 10. 리액티브 Repository 작성

```java
/**
 * 직원 리포지토리 (Spring Data R2DBC)
 *
 * - JPA Repository와 유사하지만 반환 타입이 Mono/Flux
 * - 복잡한 쿼리는 @Query로 직접 SQL 작성
 * - QueryDSL 대신 R2DBC DatabaseClient 사용
 */
@Repository
public interface EmployeeRepository extends ReactiveCrudRepository<Employee, Long> {

    /**
     * 삭제여부로 전체 직원 조회
     * - Flux: 여러 건 반환
     */
    Flux<Employee> findAllByDeletedYn(String deletedYn);

    /**
     * 부서 ID로 직원 목록 조회
     */
    Flux<Employee> findByDeptIdAndDeletedYn(Long deptId, String deletedYn);

    /**
     * 사번 존재 여부 확인
     * - Mono<Boolean>: 단일 boolean 반환
     */
    Mono<Boolean> existsByEmpNo(String empNo);

    /**
     * 복잡한 검색 쿼리 (SQL 직접 작성)
     * - :keyword 형태로 파라미터 바인딩
     */
    @Query("""
        SELECT e.*
        FROM hr_employee e
        JOIN hr_dept d ON e.dept_id = d.dept_id
        WHERE e.deleted_yn = 'N'
          AND (:deptId IS NULL OR e.dept_id = :deptId)
          AND (:status IS NULL OR e.emp_status = :status)
          AND (:keyword IS NULL OR e.emp_name LIKE CONCAT('%', :keyword, '%'))
        ORDER BY e.emp_name
        LIMIT :size OFFSET :offset
        """)
    Flux<Employee> searchEmployees(
        @Param("deptId") Long deptId,
        @Param("status") String status,
        @Param("keyword") String keyword,
        @Param("size") int size,
        @Param("offset") long offset
    );
}
```

## 11. Entity 작성 (R2DBC)

```java
/**
 * 직원 엔티티 (R2DBC 용)
 *
 * JPA와 차이점:
 * - @Entity → @Table (Spring Data R2DBC 어노테이션)
 * - @GeneratedValue 대신 @Id 만 사용 (자동 감지)
 * - 연관관계 매핑(@OneToMany 등) 미지원 → 직접 JOIN 쿼리 작성
 * - Lazy Loading 없음 (R2DBC는 별도 쿼리로 처리)
 */
@Table("hr_employee")   // R2DBC: 테이블명 매핑
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@Slf4j
public class Employee {

    @Id  // R2DBC: PK 지정 (AUTO_INCREMENT 자동 처리)
    @Column("emp_id")
    private Long empId;

    @Column("emp_no")
    private String empNo;

    @Column("emp_name")
    private String empName;

    @Column("dept_id")
    private Long deptId;          // ⚠️ R2DBC는 @ManyToOne 없음 → ID만 저장

    @Column("emp_status")
    private String empStatus;

    @Column("hire_date")
    private LocalDate hireDate;

    @Column("deleted_yn")
    private String deletedYn = "N";

    @Column("created_at")
    @CreatedDate  // Spring Data Auditing 지원
    private LocalDateTime createdAt;

    @Column("created_by")
    @CreatedBy
    private String createdBy;

    @Column("updated_at")
    @LastModifiedDate
    private LocalDateTime updatedAt;

    @Column("updated_by")
    @LastModifiedBy
    private String updatedBy;

    /**
     * 신규 직원 생성 (정적 팩토리 메서드)
     * R2DBC Entity는 new 대신 정적 팩토리 메서드 권장
     */
    public static Employee create(EmployeeRequest request) {
        Employee emp = new Employee();
        emp.empNo     = request.getEmpNo();
        emp.empName   = request.getEmpName();
        emp.deptId    = request.getDeptId();
        emp.hireDate  = request.getHireDate();
        emp.empStatus = "A"; // 기본값: 재직
        emp.deletedYn = "N";
        return emp;
    }

    /**
     * 논리 삭제 처리
     * deleted_yn = 'Y' 로 변경
     */
    public void softDelete() {
        this.deletedYn = "Y";
        log.info("[직원 논리삭제] empId={}", this.empId);
    }
}
```

## 12. 에러 처리 (전역)

```java
/**
 * 리액티브 전역 에러 핸들러
 *
 * Spring WebFlux에서는 @ControllerAdvice 대신
 * WebExceptionHandler 또는 @RestControllerAdvice 사용
 */
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    /**
     * ERP 비즈니스 예외 처리
     */
    @ExceptionHandler(ErpException.class)
    public Mono<ResponseEntity<ApiResponse<Void>>> handleErpException(ErpException e) {
        log.warn("[비즈니스 예외] code={}, message={}", e.getErrorCode().getCode(), e.getMessage());
        return Mono.just(
            ResponseEntity.badRequest()
                .body(ApiResponse.fail(e.getErrorCode()))
        );
    }

    /**
     * 입력값 검증 실패 처리
     */
    @ExceptionHandler(WebExchangeBindException.class)
    public Mono<ResponseEntity<ApiResponse<Void>>> handleValidationException(
            WebExchangeBindException e) {
        // 첫 번째 검증 오류 메시지 추출
        String message = e.getBindingResult()
            .getFieldErrors()
            .stream()
            .findFirst()
            .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
            .orElse("입력값이 올바르지 않습니다.");

        log.warn("[입력값 오류] {}", message);
        return Mono.just(
            ResponseEntity.badRequest()
                .body(ApiResponse.fail("E002", message))
        );
    }
}
```
